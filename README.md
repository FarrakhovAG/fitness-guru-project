# Fitness-Guru-project
Проект "Фитнес-Гуру". SCRUM-подход, 6-недельный цикл разработки.

Часть 1. Ключевые проблемы
1.	Отсутствие приоритетов (что в первую очередь, а что во вторую очередь)
— Команда получает сразу несколько разнонаправленных задач, причём без приоритета;
— Нет понимания, что действительно важно сейчас (монетизация, дизайн, новый функционал?).
2.	Отсутствуют пользовательские и системные требования
— Нет человека (продукт-менеджера или владельца продукта), который формулирует чёткое ТЗ и контент.
3.	Нет контроля сроков и общей цели
— Команды работают изолированно, нет общего синхронизированного процесса;
— Не настроено взаимодействие между фронтендом, бэкендом и тестированием.
4.	Отсутствие дорожной карты и сроков
— Нет общего плана релиза и контроля сроков.

Часть 2. Выбор методологии и первые шаги
1. Выбор методологии
Предложение: SCRUM. 
Обоснование: SCRUM позволит команде работать итеративно и прозрачно, расставить приоритеты через бэклог и спринты, а также наладить коммуникацию — ежедневные стендапы, ретроспективы и обзор спринта помогут команде синхронизироваться и уменьшить хаос.

2. Повестка первой встречи
Встреча: «Наведение порядка в проекте» (60 мин)
1.	Обсудить общую цель проекта и краткосрочный фокус (монетизация / релиз к выставке).
2.	Выявить текущие блокеры и зависимости (API, база данных, контент).
3.	Согласовать структуру задач и инструмент (Trello / Jira) — кто за что отвечает.
4.	Определить ближайшие приоритетные цели на 2 недели (mini-sprint).

3. Первичный бэклог (антикризисные задачи)
1.	Определить приоритетные цели проекта совместно с СЕО и зафиксировать roadmap на квартал.
2.	Собрать актуальный статус всех задач (что сделано / в работе / заблокировано).
3.	Организовать доступную и понятную доску задач (например, Trello с колонками To Do / In Progress / Done).
4.	Провести синхронизацию между разработчиками (Android ↔ Backend ↔ iOS) и решить проблему с API.
5.	Назначить ответственного за контент и функционал раздела “Тренировки”.

Часть 3. Письмо СЕО
Спасибо за письмо. Полностью согласен, что внедрение оплаты подписки — ключевой приоритет на этот квартал.
На текущий момент команда действительно сталкивается с рядом организационных трудностей: задачи пересекаются, часть работ заблокирована, а приоритеты не зафиксированы. Я уже провожу аудит состояния проекта и к концу недели представлю обновлённый план-график и список ключевых задач, необходимых для запуска подписки.
Предлагаю временно отложить второстепенные доработки (редизайн иконки и новый раздел с рецептами) и сфокусироваться на монетизации. Это позволит команде выдать стабильный результат к концу квартала и подготовиться к выставке "Спорт-2025".
Далее я представлю:
•	краткий roadmap по внедрению подписки;
•	конкретные сроки реализации;
•	прозрачную доску задач для отслеживания прогресса.
Буду держать Вас в курсе и сообщу о первых результатах уже в течение нескольких дней.

Часть 4. Диаграмма Ганта: «Внедрение оплаты подписки»

Этап	Задачи	Длительность	Зависимости
1. Анализ требований	Уточнить бизнес-логику подписки, модели (месяц/год), валюту, интеграцию с App Store/Google Play, юридические аспекты	3 дня	—
2. Дизайн UX/UI	Разработать экраны оплаты, подтверждения, ошибок; согласовать с СЕО	4 дня	После анализа
3. Backend-разработка	Реализовать API для подписки, интеграцию с платежным провайдером, хранение статусов	7 дней	После анализа
4. Frontend (Android/iOS)	Подключить API, реализовать экраны и логику подписки	6 дней	После backend и дизайна
5. Тестирование	Проверка сценариев подписки, отмены, восстановления, нагрузочные тесты	5 дней	После фронтенда
6. Релиз и публикация	Подготовка к публикации, проверка в сторе, коммуникация с маркетингом	3 дня	После тестирования

Общая длительность: 28 рабочих дней (около 6 недель).
Обоснование:
Я выделил этапы, охватывающие весь цикл разработки фичи — от проработки требований до публикации. Такая структура обеспечивает прозрачность и контроль зависимостей между командами (дизайн → backend → frontend → тестирование).
Основные риски: Имеется небольшая задержка из-за подключения к платёжному API и согласования дизайна с СЕО. Команда работает частично параллельно, а все нужные SDK и доступы к стору у нас уже есть.
